## X41 — Garde-erreur `Feuil2.Worksheet_Activate`

### Problème
Le doc-module `Feuil2` (feuille « Suivi Carburant », vit dans le `.xlsm`, non versionné) appelle `ImporterNouveauxPleinsAuto` dans `Worksheet_Activate`. Un échec (réseau, GAS indisponible) remonte et **bloque la navigation** vers la feuille.

### Solution
Enrober **l'appel** d'un garde-erreur, sur le modèle déjà en place dans `Feuil7`/Réglages :
```vba
On Error Resume Next
ImporterNouveauxPleinsAuto
On Error GoTo 0
```
Défense en profondeur pure : aucun changement de comportement nominal ; seul un import en échec devient silencieux au lieu de bloquer.

### Mise en œuvre
1. Lire le doc-module `Feuil2` live via vba-agent (COM).
2. Localiser l'appel dans `Worksheet_Activate`, l'enrober.
3. Réinjecter via vba-agent ; **test de compilation réel** (run d'une macro après injection — `set-module ok` ne prouve pas la compilation, leçon #54).

### Critère de succès
- `Worksheet_Activate` compile.
- Activer « Suivi Carburant » avec GAS injoignable n'affiche plus d'erreur bloquante.

## X51 — Projection rentabilité kit (J11/J12) filtrée par véhicule

### Problème
`J11` (date estimée de rentabilité) et `J12` (km estimé) de la feuille « Suivi Carburant » reposent sur des `LOOKUP`/`INDEX` **globaux** de `Tableau2[Date]` / `Tableau2[Km compteur]`. Depuis X50, les indicateurs (dont `B11`) sont véhicule-aware ; J11/J12 mélangent encore les pleins de tous les véhicules.

### Solution
1. Lire dans le classeur vivant la formule **véhicule-aware de `B11`** (critère = véhicule sélectionné, même mécanique que les autres indicateurs X50).
2. Décliner **les mêmes critères** sur J11 et J12, en conservant leur calcul propre (date / km de rentabilité) mais filtré par véhicule.
3. Feuille protégée → encadrer l'écriture par `DeverrouillerSuivi` / `VerrouillerSuivi` (leçon « Suivi Carburant protégée »).

### Décisions
- Reproduire **exactement** le critère véhicule de B11 (même cellule de sélection, même normalisation) — ne pas inventer une nouvelle source de filtre.
- Si la formule devient longue, respecter la limite des 25 continuations `_` (leçon VBA).

### Critère de succès
- Avec ≥ 2 véhicules, J11/J12 renvoient une date/km cohérents avec le véhicule sélectionné (vérifié en changeant la sélection).
- Formules statiques persistées dans le `.xlsm` ; feuille re-protégée après écriture.

## X43c — Rebuild dashboard incrémental (hybride)

### Problème
`modFiltres.ApplyFiltersFromControls → ScheduleRebuild → DebouncedRebuild → CreerGraphiquesWeb(silent:=True)` lance un **rebuild complet** à chaque changement de filtre. Le coût (~20-30 s) vient de `BuildAggregates` (une passe, tous les blocs). `EnsureChart` réutilise déjà les objets graphiques — l'incrémental ne porte donc QUE sur les agrégats.

### Analyse de dépendances (justifie le design)
| Filtre modifié | Blocs impactés | Incrémental utile ? |
|---|---|---|
| Véhicule (B5) | quasi tous | non (tout dépend du véhicule) |
| Période (B9/B10) | toutes séries datées | non |
| Carburant (B6) | prix/carburant + conso | partiel |
| Budget / CO2 / Année | jauges budget/CO2 | fort |

### Solution — incrémental hybride (3 niveaux)
1. **Court-circuit no-op** : calculer une **signature** de l'état de rebuild = filtres (B5/B6/B9/B10/budget/CO2/année) + **empreinte des données source** (ex. nb de lignes + max date/km de `Tableau2`+`GS_Pleins`, ou un hash léger). La stocker (cellule technique de `_GraphData` ou Name caché) après chaque rebuild réussi. Au rebuild suivant : si signature identique → **skip total**.
2. **Incrémental ciblé** : si la signature diffère **uniquement** sur des filtres isolables (carburant, budget, CO2, année) → recalculer **seulement** les blocs concernés et rafraîchir leurs charts, sans toucher aux autres.
3. **Plein** : si véhicule ou période a changé (ou empreinte données modifiée) → `BuildAggregates` complet (incompressible).

### Architecture
- Nouveau module ou helpers dans `modFiltres` / `modGraphiques` :
  - `ComputeRebuildSignature() As String` — pure, lit filtres + empreinte source.
  - `ReadLastSignature()` / `WriteLastSignature(sig)` — persistance (Name caché `_DASH_SIG` ou cellule `_GraphData`).
  - `ClassifyFilterDelta(oldSig, newSig) As RebuildScope` — enum `{ rsNone, rsTargeted, rsFull }` + liste des blocs ciblés.
- `DebouncedRebuild` consulte `ClassifyFilterDelta` :
  - `rsNone` → SetStatus « à jour » + exit.
  - `rsTargeted` → appeler un nouveau chemin `RebuildBlocks(blocks)` (recompute partiel + Add*Chart concernés).
  - `rsFull` → comportement actuel `CreerGraphiquesWeb(True)`.
- `CreerGraphiquesWeb` reste l'unique chemin « plein » ; `RebuildBlocks` réutilise les sous-`Add*Chart` existants pour ne pas dupliquer la logique de rendu.

### Risques & garde-fous
- **Risque n°1** : un blocage `rsTargeted`/`rsNone` qui laisse le dashboard désynchronisé. Garde-fou : un bouton/raccourci « rebuild forcé » (déjà existant via `RecreerDashboardComplet`) qui ignore la signature ; et invalider la signature à l'ouverture (`SyncFiltersAndRebuildOnOpen`) et après tout import de pleins.
- **Risque n°2** : empreinte source incomplète → no-op à tort après un nouvel import. Garde-fou : inclure dans l'empreinte le marqueur de dernier import (Z1 / `_SyncLog`).
- Conserver le **debounce** et le mode silencieux existants.

### Critère de succès
- Re-cliquer le même filtre → skip instantané (statut « à jour »), aucun recalcul.
- Changer uniquement le carburant → seuls les blocs prix/conso se recalculent (mesuré : nettement < rebuild complet).
- Changer le véhicule/période → rebuild complet, résultat identique à aujourd'hui.
- Un import de nouveaux pleins invalide la signature (pas de no-op à tort).

## W79 — Audit a11y CI non-bloquant (axe-core / Playwright)

### Problème
W77 a posé les aria-labels mais rien ne verrouille l'accessibilité : une régression de contraste/label passe inaperçue.

### Solution
- Dépendance dev : `@axe-core/playwright`.
- Nouveau test `tests/e2e/a11y.spec.js` : pour chaque vue principale (saisie `#/`, stats `#/stats`, historique `#/historique`), charger la page, injecter axe-core, collecter les violations **critical/serious**.
- Mode **non bloquant** : le test **rapporte** (console + résumé) sans faire échouer le job (annoter via `test.info().annotations` / `console.warn`, ou `expect.soft`). L'objectif est la visibilité, pas le gate.
- Job CI `a11y` dans `.github/workflows/ci.yml`, séparé du gate bloquant, `continue-on-error: true`. Réutilise le serveur de preview Playwright déjà configuré.

### Décisions
- **axe-core/Playwright** (pas Lighthouse) : réutilise l'infra Playwright existante, plus rapide, ciblé a11y.
- Non bloquant pour ne pas casser la CI sur des violations préexistantes ; durcissable plus tard (passer à bloquant une fois à zéro violation).

### Critère de succès
- `npm run test:e2e` (ou job dédié) produit un rapport de violations a11y sur les 3 vues.
- Le job CI `a11y` tourne, publie le résumé, et **ne bloque pas** le merge.
