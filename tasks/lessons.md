| 2026-06-10 | Plan X39 figeait "Chronologie non scriptable (Add2 xlTimeline -> Err 5 en VBA) => handoff manuel". | 2026-06-10 | Date du dernier plein affichee 06/09/2026 (6 sept) au lieu de 09/06/2026 (9 juin) dans les graphiques. `ImporterNouveauxPleins` faisait `ParseGoogleDate(CStr(cell.value))` sur une cellule `_ImportGS!Date` qui contient une VRAIE date. `CStr` → "09/06/2026" (locale FR) ; `ParseGoogleDate`, croyant recevoir du M/J/A US, lit jour<=12 comme un mois → 6 sept. Seules les NOUVELLES lignes (non encore dedupliquees) etaient touchees. | Ne JAMAIS `CStr()` une vraie date pour la re-parser : CStr est locale-dependant. Lire la date reelle : `If IsDate(cell.value) Then CDate(cell.value)` (le serial est locale-independant). L'heuristique « 1er nombre <= 12 ⇒ mois (US) » est FAUSSE pour des dates FR jj/mm/aaaa (jour <= 12). Pour ecrire une date via pywin32 sans decalage de fuseau (-2h), utiliser le **serial Excel** (`cell.Value2 = (date - 1899-12-30).days`), pas un `datetime` naif. |
| 2026-06-11 | `TimelineState.SetFilterDateRange d1, d2` sans effet (no-op silencieux) quand une borne porte une HEURE (dernier plein a 02:00) — le calage par defaut de la chronologie (`ApplyDefaultPeriodOnce`) ne « tenait » pas. | Pour caler une chronologie sur une plage, **arrondir les bornes au jour plein** (`Int(serial)`) avant `SetFilterDateRange` : une borne fractionnaire (date+heure) est ignoree. Enrober en plus la sequence de calage de `Application.EnableEvents = False` : sinon `RecreerDashboardComplet` declenche `Workbook_SheetPivotTableUpdate` → `ApplyFiltersFromControls` qui **re-applique l'ancien filtre** et annule le calage. Isoler via un helper qui lit l'etat SANS rebuild pour distinguer « SetFilterDateRange ne colle pas » de « le rebuild reverte ». |
| 2026-06-11 | Fichiers vides parasites a la racine (`code`, `$null`, `corrige`, `ui-coder`, `reconstruire`...) ramasses par `git add -A` (commit.sh) et parfois committes. `code` venait du **message de commit** contenant `design->code` : le `>code` interprete comme **redirection shell** par un hook PowerShell (PostToolUse) cree un fichier vide `code`. `$null` vient des hooks `git ... 2>$null` mal contextualises. | NE JAMAIS mettre de caracteres de redirection shell (`>`, `<`, `\|`, `&`, `2>`) dans un message de commit ou une description : ecrire « design vers code », pas « design->code ». Gitignorer les noms parasites connus. Avant `commit.sh`, faire un sweep des fichiers 0-octet a la racine (`git ls-files --others --exclude-standard` + test `! -s`) et les supprimer. Cause cote hooks : `.claude/settings.json` (Stop + PostToolUse) utilise `2>$null`/redirections — a durcir si recurrence. |
